# orchestrator (main)

The natural-language orchestrator: a long-lived Claude Agent SDK `query()` running in the main process whose only tools are an in-process `canvas` MCP server. The user talks to it from the chat bar; it drives the app (focus canvases, spawn/rename/kill cards, message agents, open/navigate/see/operate browser cards, clear permission asks) through a small command bus, narrating each action in a voice that's also spoken via TTS. Read-only tools auto-run; mutating tools pass through a host gate.

This directory also hosts the **browser backend** that two consumers share: the orchestrator's in-process MCP (above) and the **agents'** browser MCP (agentBrowserMcp.ts) — one `CommandBus`, driven through a Tier-B CDP path (browserController.ts) with the Tier-A renderer dispatch as fallback.

## Files

- **orchestrator.ts** — the loop. `runOrchestrator()` runs the SDK `query()` in streaming-input mode, holds the system prompt, wires the `canvas` MCP server, gates tools via `canUseTool`, and translates SDK messages into `OrchestratorEvent`s (streaming text deltas + tool/result/error).
- **contract.ts** — the `CommandBus` interface plus the LLM-facing `World` projection types. The seam between the SDK tools and the app; two implementations satisfy it (mainBus, stubBus). `openCanvas()` is the odd one out — it returns a plain text snapshot (not a `World` projection) of the canvas in the viewport, injected into every turn rather than exposed as a tool.
- **canvasServer.ts** — `buildCanvasServer(bus)` builds the in-process MCP server. Each tool wraps one `CommandBus` method; tools surface to the model as `mcp__canvas__<name>`. The browser tools here (`browser_read`/`screenshot`/`click`/`type`/`scroll`/`select`/`history`) take an explicit `cardId` (any browser card from `list_world`). Exports `READ_ONLY_TOOLS` (the single source for which tools auto-run) — now includes `browser_read` + `browser_screenshot`.
- **agentBrowserMcp.ts** — the **agent**-facing browser MCP: a token-scoped loopback HTTP server (`StreamableHTTPServerTransport`, stateless — fresh server+transport per POST) attached to every supervised `claude` card via `--mcp-config`, so each agent can see/control ITS OWN browser. HTTP, not in-process, because the CLI is a separate process. The calling card comes from the `X-Canvas-Card` header (`$CANVAS_CARD_ID` substituted by the CLI). One idempotent `request_browser(reason, url?)` resolves the caller's owned browser (`getState` → a browser card whose `ownerId === cardId`), else opens one linked to it and waits via `ensureReady`; the rest are the drive tools — no card id (always the caller's own), and **no discovery** (an agent only ever sees its own browser). Effects flow through the same `CommandBus` as the orchestrator. Mirrors the hook sink's port-stability dance.
- **agentIssueMcp.ts** — the **agent**-facing ISSUE MCP: the same token-scoped loopback HTTP shape as agentBrowserMcp (stateless per POST, card id from `X-Canvas-Card`, port-stability dance), attached per card via `--mcp-config` — but it talks **directly to the main `IssueStore`** (no command bus; main is the single arbiter, so a write is the same atomic check-then-append a human's click is). **Role-gated** (planner/lead/worker, resolved from the card's `role` in `getState`) and **canvas-scoped**: `get_vision` + read tools for all; a **worker** sees ONLY its own lead-assigned issues and advances/reports them — it never self-claims (the lead assigns); a **planner** gets `create_sprint` / `create_plan` / `approve_plan` (self-audit → deliver); a **lead** gets `create_issue` / `set_deps` / `assign_issue` / `request_workers` / `set_sprint_state`; planner+lead share `list_sprints`; `get_plan` (a plan's full blueprint — the planner→lead handoff) is planner/lead only. A **strategist** (the autonomous head) is read-only over the chain (`get_vision` + `get_vision_history` + `list_sprints`) and writes ONLY its own **Conception** — `record_conception` / `set_conception_winner` / `abstain_conception` (the recorded idea tournament); it has no `get_plan` and no sprint/plan/issue tools (it delivers an idea; the planner makes the sprint). Auditing is a per-role self-audit (each role spawns its own adversarial subagents before delivering), not a tool here.
- **browserController.ts** — main's owner of browser-card readiness AND the Tier-B CDP driver (`implements BrowserDriver`). (1) Readiness: turns the renderer's `browser-ready` IPC (`markReady`/`markGone`) into an awaitable `ensureReady(cardId)`, replacing a fixed settle delay; a `wake` callback nudges the renderer to re-mount a dormant/evicted browser so a wait can't hang forever. (2) Driving over `webContents.debugger` (CDP): `read` via `Runtime.evaluate` (reusing the shared in-page driver from `../../shared/browserDriver`), `act` via in-page ref→coords resolve + real `Input.*` events (trusted, work while backgrounded), `screenshot` via `Page.captureScreenshot`. Throws when CDP is unavailable so the bus falls back to Tier A.
- **mainBus.ts** — the live `CommandBus`. Projects the latest `RemoteState` for `list_world`, renders the `openCanvas()` text snapshot (the active canvas's name/branch/dirty, its cards with kind/status/task/url/running, blocked asks, and a name+attention index of the other canvases), dispatches renderer-owned mutations over a correlation-id round-trip, applies main-owned actions (agent I/O, ask decisions) directly, and fires the "comet" so each effect lands with its narration. Also defines the `BrowserDriver` interface and the `drive()` helper that routes browser ops (read/act/screenshot) through Tier-B CDP with the Tier-A renderer dispatch as fallback.
- **manager.ts** — `Orchestrator` class: owns the persistent streaming session, the input queue, the renderer command channel (`dispatch` + pending map), autonomy mode, and the IPC emit/voice tap. The main-process driver around the loop. Its `gate` builds the human-facing confirm copy here via `describeGate` (resolving canvas/card/ask ids to names from the latest `getState()`) and ships a ready-to-display `{ title, detail }` in the `confirm` command — so the renderer never reverse-engineers what a tool verb means (keep `describeGate`'s verb cases in sync with the tools registered in canvasServer.ts, the same soft contract as `READ_ONLY_TOOLS`). Takes a `browser` (`BrowserDriver`) dep it threads into the bus, and exposes a `commandBus` getter so the host can share the one live bus with the agent browser MCP server.
- **stubBus.ts** — in-memory `CommandBus` double seeding two canvases and a blocked agent, for the offline harness. No Electron.
- **harness.ts** — standalone tsx CLI (`npm run orchestrator:harness -- "<prompt>"`) that runs the loop against the stub bus with a stdin y/N confirm gate. Sanity-checks the wiring without launching the app.

## Architecture / data flow

A chat prompt enters via `Orchestrator.run()`, which enqueues an `SDKUserMessage` and wakes the input generator. That generator (`input()`) feeds the one long-lived `query()`; the session stays open between turns, idling until the next enqueue. Every yielded input message — chat prompts and `[fleet event]` heartbeats alike — fires a `UserPromptSubmit` hook that calls `bus.openCanvas()` and returns its text as `additionalContext`, so each turn is re-grounded on the canvas in the viewport (its cards, statuses, tasks, blocked asks, and real ids) plus a thin name index of the other canvases. The model operates on what's on screen with the ids shown — no `list_world` round-trip — and only reaches for `list_world` to inspect a different canvas or act on one not currently open. The model narrates, then calls a `canvas` tool. `canUseTool` first awaits `beforeTool` (speech-pacing — holds the tool until its narration has been spoken), then either auto-allows (read-only) or asks the `gate`. Allowed tool calls run against the `CommandBus`.

In production the bus is **mainBus**. `list_world` reads from the cached `RemoteState`. Renderer-owned mutations (focus/spawn/rename/kill, plus browser open/navigate) go through `deps.dispatch`, which sends an `orchestrator-command` IPC with a correlation id and awaits the renderer's reply via the pending map (30s timeout → failure result). Main-owned actions (`send_to_agent` via keystroke injection, `get_agent_reply`, `approve_ask`) bypass the renderer entirely. Every mutation fires `signalTarget` (a comet from chat bar to card) and `await landed()` so the effect commits when the comet arrives.

**Browser cards** are a third card kind: in-app web views with no agent/tmux session, so the model can't `send_to_agent`/`get_agent_reply` them (the system prompt says so), but it can now both navigate and *operate* them. `open_browser` (renderer `spawnBrowser`) creates one, optionally at a starting url; `navigate_browser` (renderer `navigateBrowser`) points an existing one at a new url — mainBus guards `kind === 'browser'` and signals/lands the comet before dispatching. Beyond url, the model sees a browser's page through `browser_read` (a set-of-marks snapshot: interactive elements with stable `ref`s + page text) and `browser_screenshot`, and acts via `browser_click`/`type`/`scroll`/`select`/`history` keyed on those refs. The loop is read → act → re-read (refs are only valid for the latest read). `list_world` still projects each browser's current `url` (name reflects the page title).

**Browser see-and-control: two consumers, one backend.** The orchestrator reaches the page via the canvas MCP tools above (explicit `cardId`); each **agent** reaches *its own* page via agentBrowserMcp.ts (no card id, resolved from `ownerId`). Both land on the same `CommandBus` methods — `readBrowser`/`screenshotBrowser`/`actBrowser`/`setBrowserReason`. In mainBus those three observe/act methods run through `drive()`: try **Tier-B** (browserController over CDP — real input, works while the card is stacked/backgrounded), and on a thrown CDP failure fall back to **Tier-A** (dispatch to the renderer's in-page driver). `actBrowser` flies a comet + `landed()` like a mutation; `readBrowser`/`screenshotBrowser` skip it (the agent loop reads frequently). `setBrowserReason` and `open_browser`'s `ownerCardId`/`reason` thread an agent's provenance onto the browser card's window bar. Spawning a browser no longer sleeps a fixed delay before the first read: `request_browser` (and the Tier-B `cdp()` helper) await `browserController.ensureReady`, gated on the renderer's `browser-ready` signal.

Two other producers push into the same queue as heartbeats — nothing is polled: `notifyAgentReply()` (an agent's Stop hook fired) and `notifyAsk()` (an agent hit a permission request). Both inject `[fleet event]` messages for awareness only; the system prompt forbids acting on them without a standing user instruction.

Outbound, `runOrchestrator`'s `onEvent` → `Orchestrator.emit()` sends `orchestrator-event` IPC to the renderer chat and taps `deps.speak` for TTS.

## Conventions & gotchas

- **Auth / token source**: the loop deletes `ANTHROPIC_API_KEY` at startup so a stray key can't outrank the subscription and silently bill pay-as-you-go. The SDK then uses `CLAUDE_CODE_OAUTH_TOKEN` if exported, else the host's stored `claude login` creds. No token export needed when signed into Claude Code. Model is pinned to `claude-opus-4-8`.
- **Built-in tools are dropped** (`tools: []`): the orchestrator only has the canvas verbs, never file/build/repo access. The system prompt repeats this — it operates the app, it is not a coding agent.
- **Read-only single source**: `READ_ONLY_TOOLS` in canvasServer.ts feeds the loop's `READ_ONLY` set (which tools `canUseTool` auto-allows) and the `readOnlyHint` annotations. Keep the two in sync. Note there is **no `allowedTools`**: listing reads there would auto-approve them *before* `canUseTool` runs (the SDK's permission order is allow-rules → `canUseTool`), skipping the speech-pacing in `beforeTool`. Routing every tool — reads included — through `canUseTool` is what makes pacing cover them; that callback is the single point for both permission and ordering.
- **Tier-B first, Tier-A fallback, never silently**: browser observe/act go through `drive()` (CDP, else renderer). The transport is logged only on change, so a silent fallback can't masquerade as CDP. `CANVAS_BROWSER_STRICT_CDP=1` makes a CDP failure throw instead of falling back — the verification mode to confirm CDP actually works; off by default, because the fallback is real production resilience. The boundary: a thrown error from the driver means CDP is unavailable (no live guest, can't attach, DevTools holds the session) → fall back; a logical `{ ok:false }` (e.g. a stale ref) is a normal result, not a throw.
- **Agents see only their own browser**: agentBrowserMcp.ts has no discovery tool and the drive tools take no card id — ownership is resolved by `ownerId === callerCardId` from `getState()`. `request_browser` is idempotent (returns the owned browser, updating its reason) so repeated calls don't pile up browser cards. The orchestrator's canvas tools, by contrast, take an explicit `cardId` and can touch any browser. Both share one `CommandBus` (the manager's `commandBus` getter hands the same instance to both servers).
- **Barge-in / interrupt**: `runOrchestrator` hands the live query handle up via `onSession`; `Orchestrator.interrupt()` calls it to stop the current turn. Wired to the mic: `voice-stt-start` in main fires `voice.cancelSpeak()` (drop the audio) **and** `orchestrator.interrupt()` (stop the turn) so a multi-block turn can't start a fresh spoken line a beat after you grab the mic. `interrupt()` is gated on `turnActive` (tracked off the event stream) so a barge-in while idle is a no-op. The interrupted turn closes with a non-success `result` (the SDK reports an aborted turn as `error_during_execution`); `emit` sets an `interrupted` latch so that one expected error is swallowed instead of whispered/spoken as a red failure.
- **Modes** (`manual` | `partner` | `autonomous`) — how a sprint is BORN, not an oversight dial. `manual` gates every orchestrator action (5-min human confirm) and suppresses fleet-event/milestone wakes entirely. `partner` and `autonomous` both let the orchestrator's own tools run freely and wake it on events; they share one execution cascade (planner → lead → workers, each self-auditing) and differ only at the head: in `partner` the human originates work via a planner interview and confirms the plan; in `autonomous` the mastermind originates it itself via the **strategist** (a card that runs the pinned idea tournament; see the cascade below). Default is `manual`.
- **Mastermind cascade** — in partner/autonomous, the orchestrator drives role cards. `notifyMilestone` (wired from `IssueStore.onMilestone`) injects a `[fleet event] PLAN READY` when a sprint's plan is approved → the orchestrator spawns a `lead` (the system prompt's MASTERMIND CASCADE section). The lead hires its own workers via the issue MCP's `request_workers`, which calls `Orchestrator.requestWorkers` — that spawns stand-by `worker` cards (clamped ≤8) on the lead's canvas and returns their ids for `assign_issue`. The orchestrator audits nothing; it reacts only to milestones and trusts the flow. The cascade is **event-driven, no polling** — the mastermind is the routing hub for cross-card notifications: `assign_issue` → `issue-assigned` → nudge the worker (kills the spawn/assign race); a worker's `done`/`blocked` → `issue-done`/`issue-blocked` → nudge the **lead** (assign the next frontier / unstick it); and `notifyAsk` auto-approves permission asks from **role cards** (planner/lead/worker) so it runs unattended — a plain agent's asks still surface to the human. Every nudge is a deterministic `sendToAgent` keyed off a store state-change, not the model parsing reply text. The planner can `create_sprint` itself (from a partner interview), so the human only needs to set the canvas vision, not pre-create the sprint. **Role cards boot straight into role**: `mainBus.spawnAgent` deterministically leads a role card's initial prompt with `/canvas-skills:mastermind-<role> <mode>` (mode from `getMode`: partner vs autonomous) — a slash command at the start of the initial prompt runs on turn 0, so the card invokes its skill before anything else, no reliance on auto-discovery. This is the single chokepoint, so it covers both LLM-spawned (planner/lead) and `requestWorkers`/`spawnStrategist`-spawned cards. **Autonomous head (strategist)**: in autonomous mode the mastermind spawns a `strategist` card when a canvas goes idle — on entering autonomous (a one-shot check) and on each `outcome-verified` (a sprint reaching DONE) — guarded against double-spawn and against running while a sprint is in flight. The strategist runs its pinned tournament and records a Conception; `idea-ready` (a winner) → the mastermind **directly** spawns an unattended `planner` seeded with the winning idea verbatim (then the partner cascade runs); `idea-abstained` (no winner) → escalate to the human via a fleet event, never manufacture a sprint.
- **stub vs main bus**: stubBus is offline-only (harness); mainBus is the real seam. They diverge in behavior — e.g. stub's `send_to_agent` just echoes, main injects keystrokes and queues when busy; the stub's browser methods return canned snapshots/results (no CDP, no renderer). Both implement `openCanvas()` in the same shape (keyed off `WorldCanvas.active` in main, the `active` id in stub), so the harness re-grounds each turn too.
- **Per-turn re-grounding vs `list_world`**: the open-canvas snapshot is *injected*, not a tool, so it costs no tool call and can't be skipped — but it only covers the viewport canvas. Acting on another canvas still requires `list_world` to learn its card ids. This split is a soft convention enforced only by the system prompt and the snapshot's `list_world for their cards` hint, not by code; the model can still call `list_world` whenever it wants.
- **Relative imports, not `@shared`**: contract.ts and siblings import shared types by relative path so the standalone tsx harness resolves without tsconfig-path support.
- **Speech-pacing lives in `beforeTool`/`canUseTool`, not the gate** — it covers every tool and is the single mechanism ordering voice and action. The gate only decides permission.
- The input generator must never throw — a throwing generator aborts the SDK session opaquely. The session auto-restarts in `startSession`'s finally if work is still queued.
- Streaming text is emitted token-by-token (`includePartialMessages`); a per-turn `streamed` latch falls back to the complete text block if partials never arrive (older CLI / cached turn).
