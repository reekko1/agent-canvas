# spine (main)

The spine runs each agent card as a **headless session** — one long-lived Claude Agent SDK `query()` per claude card, a turn-batched `codex exec --json` subprocess per codex card — and turns whatever each CLI emits into `CardEvent`s, transcript entries, and captured replies. There is no tmux, no pty, no hooks: a card's process IS the driver's own SDK session or subprocess, nothing sits in front of it translating a wire protocol. It is the seam between a folder-of-work card and a live agent: it owns the driver registry, the live session map, and the transcript store. (Structured questions are NOT a spine concern — they ride the CLI-agnostic `mcp__canvas__ask_user` tool in `orchestrator/agentCanvasMcp.ts`; that's the only human-decision channel now that there's no permission-hook hold to fall back to.)

## Files

- **spine.ts** — `Spine` class, the orchestration hub. Owns the driver registry (claude/codex), the `sessions` map of live `AgentSession`s, the `TranscriptStore`, and the remote panel. Exposes `ensureAgent(cardId, folder, { cli, initialPrompt, resume })` (idempotent — registers a session slot on a card's first `startAgent`, called every CardNode mount; does NOT necessarily spawn a live process — see below), `sendToAgent`/`interruptAgent`/`killCard`/`loadTranscript`, `skillRef(cli, name)` (driver-resolved role-skill invocation), `availableClis()` (probes each driver's `binary` over the login shell), and `shutdown()` (interrupts every live session at app teardown). `handleSessionEvent` is the one place a driver's `SessionEvent` fans out: `.card` → `onUpdate` (the existing `card-event` channel, unchanged), `.item` → persisted (if not `streaming`) + `onTranscriptItem`, `.reply` → captured into `replies` + `onReply`. Holds `SPINE_DIR` (`~/.agentcanvas-web`) and the persisted `SpineConfig` (token + `remotePort` + the `mcpPorts` registry — `sinkPort` is gone, there's no hook sink anymore).
- **driver.ts** — the `CliDriver` / `AgentSession` / `SessionEvent` / `SessionSpec` interfaces: the transport/config/session-lifecycle contract one coding-agent CLI implements. A driver is constructed with the staging dir + spine token; `stageInstructions()` / `stageMcp(id, port, opts?)` / `skillRef(name)` mirror the old adapter shape, but `start(spec, cb)` replaces `launchCommand` — it returns a live `AgentSession` (`send`/`interrupt`/`kill`, `sessionId` once captured) instead of a shell command string. `SessionEvent` is `{ card?, item?, reply? }` — a driver emits any subset per message it reads. Also home of `McpStageOpts` and the **shared SessionEvent vocabulary** — `sessionStarted` / `turnDone` / `turnFailed` / `interrupted` plus `clip` / `itemIdSeq` — the load-bearing event shapes the renderer's reducer keys on (session start clears the plan, a done turn clears the task line and carries the reply, an interrupt is an idle note not an error). Both mappers and both drivers compose these instead of hand-building literals, so a new CLI can't get the choreography subtly wrong.
- **claudeDriver.ts** — `ClaudeDriver`, one Agent SDK `query()` per card in streaming-input mode (the same pattern the orchestrator uses). **Lazy**: `start()` never calls `query()` itself — it only does so when there's an `initialPrompt` (a fresh spawn) or on the session's first `send()`, so a restored card with nothing pending costs nothing until you actually talk to it. `send()` pushes onto a queue+wake pair the SDK accepts mid-turn, so it's always `'sent'`. MCP servers are a typed `mcpServers` option built fresh per session, with the literal `cardId` baked into the `X-Canvas-Card` header directly — no more `${CANVAS_CARD_ID}` env-substitution hack. The always-on baseline rides `systemPrompt: { type: 'preset', preset: 'claude_code', append: BASELINE_SUPERVISION }` (no more `--append-system-prompt-file`); the role-skill plugin dir is a `plugins` entry, materialized the same way the old `ClaudeAdapter` did. `interrupt()` sets a flag then calls `query.interrupt()`; the mapped `error_during_execution` result that follows is swallowed and re-emitted as a plain `'Interrupted'` system item instead of an error (mirrors the orchestrator's own interrupted-turn handling in `manager.ts`).
- **codexDriver.ts** — `CodexDriver`, turn-batched: `codex exec --json` is spawned fresh per turn (no live process between turns), so a card's "session" is really a thread id + a small `idle`/`running` state machine with a FIFO queue for messages that arrive mid-turn (codex has no mid-turn input channel — a `send()` while `running` returns `'queued'` and is delivered, coalesced, as the next `resume` turn once the current child exits). `interrupt()` is a plain `SIGKILL` of the turn's child — verified empirically safe: codex persists its transcript incrementally, so a killed turn resumes with zero redone work. The prompt travels over the child's **stdin** (`codex exec ... -`), never interpolated into the shell command, so no quoting concern for model/user-authored text. Everything the old `CodexAdapter` did for CODEX_HOME staging is kept verbatim — the `auth.json` symlink to `~/.codex`, `AGENTS.md` baseline, the role-skill plugin marketplace + lazy `codex plugin` install, and the `config.toml` `[mcp_servers.*]` read-modify-write (`stripMcpTables`). What's gone is the hook staging (curl command-hooks, `hooks.json`, `--dangerously-bypass-hook-trust`) — `--json` emits the event stream natively, so there's nothing left to reverse-engineer from hooks.
- **claudeEvents.ts** — `ClaudeEventMapper`, a pure(-ish) per-session mapper from one Agent SDK stream message to zero or more `SessionEvent`s. Stateful only for what must span messages within a turn: accumulating streamed text (`stream_event` content-block deltas), a `tool_use_id → tool name` map (so a later `tool_result` failure can name the tool), and a `task_started`/`task_updated` counter for subagents (counter only, never status — ported rule from the old `hookEvents.ts`). Deliberately tolerant: an unrecognized SDK message type yields no events. Ports the status vocabulary of the retired `hookEvents.ts` (tool-failure-stays-running, StopFailure-equivalent semantics on a non-success `result`) onto the SDK's own message shapes.
- **codexEvents.ts** — `CodexEventMapper`, the same idea for one parsed line of `codex exec --json` JSONL. Schema facts about codex's wire format live ONLY here (verified empirically against codex 0.136.0; parsed tolerantly — an unrecognized `type`/`item.type` yields no events, since the exact shape may drift across versions). Tracks the last `agent_message` text across a turn (`turn.completed` carries only token usage, not the text — the tracked text becomes the turn's `reply`).
- **transcripts.ts** — `TranscriptStore`, append-only JSONL persistence at `SPINE_DIR/transcripts/<cardId>.jsonl`, one finalized `TranscriptItem` per line. Streaming deltas are NEVER persisted — only the closing, non-streaming item — so the file is a clean replay of what a human would actually have seen land. `load()` tails the last 500 lines and tolerates a truncated last line (a crash mid-append). `delete()` drops a card's transcript on the ✕ path.
- **ptyTitles.ts** — shell-card title helpers for a DIRECT pty (no tmux pane to query anymore). `foregroundCommand(pid)` walks the pty's direct children via `pgrep -P` + `ps -o pid,stat,command`, picking the one sitting in the foreground process group (`+` in STAT) — ported from the old `tmux.ts`'s ps-walkers, just keyed on a known pid instead of resolving one from a tmux pane. `paneCwd(pid)` reads the cwd via `lsof -a -p <pid> -d cwd -Fn` (there's no `#{pane_current_path}` to read anymore).
- **instructions.ts** — unchanged in mechanism: `BASELINE_SUPERVISION` (the always-on supervision briefing) and `CanvasSkill` + `materializeSkill` (the SKILL.md format spec and serializer). Each driver still renders the baseline in its own always-on channel — claude via `systemPrompt.append` (was `--append-system-prompt-file`), codex via `AGENTS.md` — and still materializes the whole Mastermind ROLE library (`../mastermind/roleSkills.ts`) into its own plugin form. The baseline text itself was rewritten: no more tmux/scrollback claim (a headless session doesn't survive a quit — the transcript does, and that's what's promised instead), and the permission-asks paragraph now just states cards run unattended, full stop (no more "if your session is in a gated permission mode" branch — that mode doesn't exist anymore).

**Deleted this rewrite**: `tmux.ts`, `hookSink.ts`, `cliAdapter.ts`, `claudeAdapter.ts`, `codexAdapter.ts`, `hookEvents.ts`. Also gone: the whole spine held-permission-ask model (`HeldAsk`, `asks`/`askSeq`, `decide`, `releaseFor`, `onAsk`) — headless bypass-mode cards never emit a permission hold to begin with (claude's `PermissionRequest` hook doesn't even fire in `-p`/SDK mode; codex's never did under `--ask-for-approval never`), so there was nothing left to hold.

## Architecture / data flow

A card's agent session is registered, not launched, the moment its `CardNode` mounts: the renderer calls `startAgent` (→ `spine.ensureAgent`) unconditionally on every mount, restored or fresh. `ensureAgent` is idempotent and — critically — **lazy**: it always constructs an `AgentSession` object via the resolved driver's `start()`, but neither driver necessarily starts a real process doing so. Codex is lazy by construction (nothing to start until a turn is requested); claude's SDK `query()` is *made* lazy the same way (deferred until an `initialPrompt` or the first `send()`). So a relaunched app with a dozen restored cards costs nothing until you actually talk to one of them — this is the mitigation for the SDK-session-per-card footprint risk.

A driver's `start(spec, cb)` is given `{ cardId, folder, resume?, initialPrompt? }` and a pair of callbacks (`onEvent`, `onExit`); everything the driver reads from its CLI (SDK stream messages for claude, JSONL lines for codex) goes through its `*EventMapper` and comes out the other side as `SessionEvent`s, handed to `cb.onEvent`. `Spine.handleSessionEvent` is the one fan-out point: `.card` → the existing `card-event` IPC channel (unchanged shape — `CardEvent` didn't need to change for this rewrite), `.item` → the transcript (persisted if finalized, always pushed live via `onTranscriptItem` → `transcript-item` IPC), `.reply` → `spine.replies` + `onReply` (the orchestrator's `get_agent_reply` read path, unchanged). `cb.onExit(reason, detail)` fires when a session's process/loop ends on its own — `Spine.handleSessionExit` drops it from `sessions` and fires `onSessionEnded` (→ `session-ended` IPC, the agent-card analogue of a shell's `pty-exit`).

**Restart semantics — sessions do NOT survive an app quit.** This is a deliberate product principle, not a limitation: an agent shouldn't keep executing once nobody is supervising it. `Spine.shutdown()` (called from `index.ts`'s `before-quit`) interrupts every live session — fire-and-forget, since both drivers persist their transcript incrementally, so at worst the in-flight tool call is lost, not the conversation. What DOES survive: the JSONL transcript (`TranscriptStore`) and, in `CardRecord.session` (persisted via `useWorkspace`), the CLI's own session/thread id. On relaunch, a card's transcript repaints from disk and it sits idle; the first `sendToAgent` after that resumes the SAME conversation by passing that id as `SessionSpec.resume` (claude: SDK `resume`; codex: `resume <threadId>` before the trailing `-`). A stale/invalid resume id just falls back to a fresh session on the next attempt — harmless.

Config/identity isolation is unchanged: dir `~/.agentcanvas-web` — side-by-side with the shipping Swift app until cutover, when this becomes `~/.agentcanvas`.

## Conventions & gotchas

- **The transcript's upsert-by-id protocol is load-bearing.** A streaming assistant message re-pushes under the SAME `TranscriptItem.id` with growing text (`streaming: true`) until it closes (one final push, `streaming` absent) — the renderer replaces-by-id rather than appending, and `TranscriptStore` only ever persists the finalized push. This means `load()` + the live `onTranscriptItem` feed can overlap (a push arriving mid-`loadTranscript`) with no duplicate or stale-looking rows: last-wins-by-id on the renderer side handles it.
- **Only claude streams assistant text; codex lands it whole — a transport limit, not a bug.** Claude's Agent SDK emits `stream_event` content-block deltas (`includePartialMessages: true`), so `claudeEvents` re-pushes a growing `streaming` item token-by-token. `codex exec --json` has NO delta/partial event: per the [Non-interactive Mode docs](https://developers.openai.com/codex/cli/reference) the assistant text appears ONLY in a single terminal `item.completed` (`agent_message`) — stdout gets "only the final agent message". So `codexEvents.mapItemCompleted` pushes it in one chunk; there is no partial to map. Token-streaming codex would mean abandoning `codex exec` for its App Server / MCP Server transport (a persistent server, not turn-batched subprocesses) — which would also forfeit the SIGKILL-a-turn-and-resume-with-zero-redone-work property. Deliberately not done; the card shows `running` from `turn.started` so codex still reads as live while it works.
- **`ensureAgent` records the driver at registration time**, not at every `send`. `setCardCli` is also seeded from the persisted workspace on `load-workspace` — so a restored card's first `startAgent` resolves the right driver even before any session event has arrived.
- **Every card runs unattended by default**, unconditionally now — not "by default, with a flip". Claude: `permissionMode: 'bypassPermissions'` + `allowDangerouslySkipPermissions: true`. Codex: `--ask-for-approval never`. There is no gated mode, no shift+tab flip, no held permission ask to toast — the only human-decision channel left is the canvas MCP's `ask_user` (a `q-<n>` id, held entirely inside `agentCanvasMcp.ts`, nothing to do with the spine).
- **Subagent events touch the counter only, never status** — still true: `task_started`/`task_updated` (claude), never mapped to a status change, since they fire out of sync with the turn's own `result`/`turn.completed` and would flip a finished card back to running.
- **Tool failures stay `running`** (a failing test is the work); only a non-success turn end (`result.subtype !== 'success'` for claude, `turn.failed`/`error` for codex) produces `stalled`/`error` — the guard against a card glowing "running" forever.
- **The plan is in-memory, not on disk.** Unchanged: the checklist is driven by the canvas MCP `update_plan` tool (native `TodoWrite`/`TaskCreate` are disabled via `disallowedTools`/absent-by-default), which emits a `todoChange` `CardEvent`. A relaunched card shows no checklist until its next `update_plan` — the plan doesn't survive an app restart, same v1 tradeoff as before, just for a different underlying reason (no live process to have kept it in memory).
- **`interrupt()` semantics differ by driver, and callers shouldn't care which.** Claude: `query.interrupt()` — a real API-level abort, the session's input generator stays alive for the next `send()`. Codex: `SIGKILL` the turn's child process — courser-grained, but verified empirically to lose nothing but the interrupted tool call (both CLIs write their transcript as they go, not just at turn end).
- **`stageMcp` for claude writes nothing to disk anymore.** It just records `{port, opts}` on the driver instance; the actual `mcpServers` option is built fresh inside `start()` from whatever's been staged so far, per-card, with the real `cardId`. Codex still writes `config.toml` (a real subprocess with real env vars — `env_http_headers` reads `CANVAS_CARD_ID` from the child's env, which the driver always sets).
