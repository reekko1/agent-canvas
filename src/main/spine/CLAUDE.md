# spine (main)

The spine launches and reattaches real coding-agent CLI sessions (`claude` or `codex`) inside tmux and ingests hook events ‚Äî the canonical schema Claude Code defined and codex adopted ‚Äî to drive each card's status, checklist, and final reply. It is the seam between a folder-of-work card and a live agent: it owns the tmux launch path, the loopback hook sink, the per-CLI adapters, and the held permission/question asks. It is a faithful port of the Swift Spine; the event-mapping facts were empirically verified there against real hook payloads ‚Äî trust the comments.

## Files

- **spine.ts** ‚Äî `Spine` class, the orchestration hub. Owns the sink/adapters/tmux/remote, maps `cardId` ‚Üî tmux session name, exposes `launch`/`kill`/`decide`/`answerQuestion`/`lastReply`, and routes every incoming hook in `handle()` through the card's adapter (recorded per card at `launch`, or re-seeded via `setCardCli` for a session that survived an app restart). Holds `SPINE_DIR` (`~/.agentcanvas-web`) and the persisted `SpineConfig` (token + sink/remote ports + the `mcpPorts` registry ‚Äî `loadConfig` migrates the old per-server `browserMcpPort`/`issueMcpPort` fields into that registry on first read). Exposes a `token` getter, `mcpPort(id)` / `attachMcp(id, port, opts?)` so the agent-facing MCP servers (in orchestrator/) share the token and persist their ports ‚Äî cards authenticate to them exactly as their hooks do ‚Äî `skillRef(cli, name)`, the adapter-resolved role-skill invocation the orchestrator leads role prompts with, and `availableClis()`, which probes each adapter's `binary` over the login shell (`command -v`) so the spawn UI only offers CLIs actually on PATH.
- **cliAdapter.ts** ‚Äî the `CliAdapter` interface: the transport/config/launch + event-mapping contract one coding-agent CLI implements. Adapters are constructed with the staging dir + spine token (the constants of a spine's lifetime), so the methods carry only what varies: `stageHooks(port)` / `stageSkills()` / `stageMcp(id, port, opts?)`, `launchCommand`, `skillRef`, and the mapping trio `classifyAsk` (permission gate vs structured question vs telemetry ‚Äî including the question's `input` round-trip payload) / `event` / `finalReply` plus the decision bodies. A CLI lacking a capability no-ops the method ‚Äî the spine degrades gracefully instead of branching per CLI. Also home of `shellQuote`, `McpStageOpts`, and `HookAsk`.
- **claudeAdapter.ts** ‚Äî `ClaudeAdapter`, the reference implementation. Writes scoped HTTP hooks to `hooks.json` (via `--settings`, never touching user config), stages the two instruction channels (`stageSkills` ‚Äî the always-on baseline to `baseline.md` folded in as `--append-system-prompt-file`, and the on-demand role library as a `--plugin-dir` plugin) and the agent-facing MCP configs (`stageMcp(id, ‚Ä¶)` ‚Üí `<id>-mcp.json`, all riding one variadic `--mcp-config`), and builds the `exec claude` command. Delegates all pure mapping to hookEvents.
- **codexAdapter.ts** ‚Äî `CodexAdapter`. Codex adopted the canonical hook schema, so the mapping in hookEvents applies wholesale; what's owned here is transport (command hooks that curl the sink ‚Äî no `--settings` equivalent, so `CODEX_HOME` is redirected to a private dir with `auth.json` symlinked back) and delivery (baseline via `AGENTS.md`, role skills via a local plugin marketplace, MCP servers accumulated into `config.toml` by `stageMcp`).
- **skills.ts** ‚Äî the CLI-neutral agent instructions, authored once as pure data. Two kinds: `BASELINE_SUPERVISION`, the **always-on** supervision briefing every card boots with (each adapter renders it in its own always-on channel ‚Äî Claude `--append-system-prompt-file`, codex `AGENTS.md` ‚Äî never a skill, so auto-discovery can't skip it), and `CANVAS_SKILLS`, the **on-demand** Mastermind role library (`mastermind-planner`/`-lead`/`-worker`). Each role skill encodes that role's flow and its mandatory self-audit step (spawn adversarial subagents before handoff) and names only the CLI-agnostic canvas tools (`update_plan`/`ask_user`); a card follows the one matching the role it was spawned as (its brief names the role), and the issue-MCP tool grant enforces actual capability. `ClaudeAdapter.stageSkills` materializes the role library into a Claude Code plugin under `SPINE_DIR/canvas-skills/`. The idea tournament is no longer a card skill ‚Äî it runs **off-card** on the mastermind's own headless `claude` (`orchestrator/tournament.ts`), so it's identical for every canvas regardless of its cards' CLI (`STRATEGIST_TOURNAMENT_SRC` is still exported here, consumed there).
- **hookEvents.ts** ‚Äî pure mapping for the **canonical hook schema** (the wire format Claude Code defined and codex adopted); no fs/process/state. Maps lifecycle events to `CardEvent`, parses/answers `AskUserQuestion`, builds permission allow/deny bodies, extracts the final reply (`finalReply`). Both adapters delegate here through their own methods ‚Äî schema drift belongs in the adapter, not this file. Testable in isolation.
- **hookSink.ts** ‚Äî `HookSink`, a loopback HTTP server. Each hook POSTs JSON to `/hook` with the card id in `X-Canvas-Card`; the response body is bidirectional, so a held ask keeps the connection open until the user decides.
- **tmux.ts** ‚Äî `Tmux`, the session substrate over a canvas-owned tmux server (socket `agentcanvas-web`). Probes the binary, writes a private `tmux.conf`, builds client/attach commands, and queries pane state. Pure helpers at the bottom resolve a shell's foreground command for shell-card titles.

## Architecture / data flow

A card maps 1:1 to a tmux session named `canvas-<cardId>` (the single source of truth is `Spine.sessionName`). `launch()` runs the tmux *client* via `new-session -A`, so the same call **reattaches** a surviving session or **creates** one running the card's CLI (`claude` or `codex`, resolved per card via `setCardCli`/`adapterFor`) under the user's login shell (`-lc`) for a real PATH. This is reattach-not-resume: there is no `--resume`; the live tmux session simply outlives the app, and quitting the app only detaches the client. `killSession` (the ‚úï path) is the one place an agent's life actually ends ‚Äî detaching would leave it running headless, the unsupervised state the canvas exists to prevent. A bare-shell card launches just the login shell (no agent, no hooks). No tmux binary ‚Üí direct pty spawn that dies with the app.

Hooks arrive over HTTP. `ClaudeAdapter.stageHooks` writes `hooks.json` pointing every event at `http://127.0.0.1:<sinkPort>/hook` with the spine token in `X-Canvas-Token` and `$CANVAS_CARD_ID` (set into the tmux *session* env via `-e`) in `X-Canvas-Card`. `CodexAdapter.stageHooks` reaches the same sink over the same headers, just via a `type:"command"` hook that curls it (Codex has no native HTTP hook transport). `HookSink` authenticates the token, parses the payload, and calls `onRequest`. `Spine.handle` then branches three ways: an `AskUserQuestion` (which rides the `PermissionRequest` channel) is siphoned to `onQuestion`; a real `PermissionRequest` goes to `onAsk`, held open as the decision channel; everything else is acked immediately (`respond(null)` ‚Äî telemetry never blocks the agent) and mapped to a `CardEvent` emitted via `onUpdate`. A turn-ending event's full reply (the adapter's `finalReply` ‚Äî the spine knows no CLI's event names) is also captured into `replies` and surfaced via `onReply`/`lastReply` for the orchestrator's get-reply path. Held asks are resolved by id through `decide`/`answerQuestion`, or released en masse via `releaseFor` when the terminal is focused.

Config/socket isolation is deliberate: dir `~/.agentcanvas-web`, tmux socket `agentcanvas-web` ‚Äî side-by-side with the shipping Swift app until cutover, when these become `~/.agentcanvas` / `agentcanvas`.

## Conventions & gotchas

- **Spine identity must survive restarts.** tmux sessions outlive the app and read their hook URL + token *once* at launch. So the sink retries the **same** persisted port for ~10s before conceding to ephemeral (a dev hot-restart overlaps the dying process); a fresh port or token would strand every surviving agent on a dead URL. `spine.json` and `hooks.json` are `chmod 0600` ‚Äî both carry the token.
- **`stageHooks` runs only after the port binds**, inside the sink's `onReady`. Cards spawn lazily, long after; if the sink isn't ready `launchCommand` falls back to bare `exec claude`.
- **Skills are equipped, not selected.** `stageSkills` runs up front in `Spine.start()` (no sink/port dependency) and rebuilds `SPINE_DIR/canvas-skills/` + `baseline.md` from scratch each startup, so editing `skills.ts` ships on relaunch and a removed skill never lingers. The *whole* role library attaches to *every* agent card ‚Äî there is no per-card subset (a deliberate v1 non-goal); the always-on baseline rides `--append-system-prompt-file` alongside it. Instructions are authored as in-process data (not on-disk SKILL.md behind `extraResources`) specifically to avoid dev-vs-packaged path resolution; the materializer is source-agnostic if that changes later. Descriptions are emitted as JSON-quoted YAML scalars so a `:` in the text can't break frontmatter.
- **The agent-facing MCP servers live in orchestrator/, not here.** The spine only owns their *token* and *stable ports* and stages the per-card config ‚Äî exactly parallel to how it stages skills and hooks, and generic over server id (`browser` / `issues` / `canvas`). `index.ts` constructs each server with `spine.token`, starts it on the persisted `spine.mcpPort(id)`, and its onReady calls `attachMcp(id, port, opts?)` (mirroring the sink's `onReady`), which persists the port in the `mcpPorts` registry and runs every adapter's `stageMcp`. The id is the server key ‚Äî it becomes the card's tool namespace (`mcp__<id>__*`) ‚Äî and `opts` carries the server's declared needs (the canvas server's `ask_user` blocks on a human ‚Üí a long `toolTimeoutSec`). Ports are persisted for the **same reason as the sink's**: surviving tmux sessions read their staged MCP config once at launch, so a fresh port would strand them. `stageMcp` can only run once a server binds, so a card launched before it simply lacks that server's tools. The staged `<id>-mcp.json` files are `chmod 0600` (they carry the token) and use `X-Canvas-Card: ${CANVAS_CARD_ID:-unknown}` ‚Äî the CLI substitutes the tmux session env the same way the hooks identify a card.
- **`--mcp-config` forces a `--` before the initial prompt.** It's variadic (`<configs...>`), so without an explicit end-of-options marker it swallows the trailing prompt as another config and `claude` dies at startup ("terminal exited"). `launchCommand` emits `exec claude <flags> -- <prompt>` whenever there's an initial prompt. No `--strict-mcp-config`: the staged servers only *add* to whatever MCP servers the user already configured globally.
- **Codex cards run unattended, by design.** `CodexAdapter.launchCommand` passes `--ask-for-approval never` (plus `--dangerously-bypass-hook-trust` so our provisioned hooks don't need a manual trust step), so codex never emits a held `PermissionRequest` ‚Äî supervision is entirely via the telemetry hooks (status/reply) and the ‚úï kill path, not an allow/deny gate. `AskUserQuestion` has no Codex equivalent either, so `classifyAsk`'s question branch never fires for a codex card; both are inert, not broken.
- **tmux session targets use `=name` (exact match)** everywhere ‚Äî a bare `-t` prefix-matches, so `card-1` would kill/query `card-10`. `display-message` additionally needs the trailing colon (`=name:`) to resolve a session's active pane.
- **Subagent events touch the counter only, never status** ‚Äî they fire out of sync with the main `Stop` and would flip a finished card back to running.
- **Tool failures stay `running`** (a failing test is the work); only `StopFailure` produces `stalled`/`error`, the guard against a card glowing "running" forever.
- **The plan is in-memory, not on disk.** The checklist is driven by the canvas MCP `update_plan` tool (native `TodoWrite`/`TaskCreate` are disabled on every card), which emits a `todoChange` `CardEvent` the renderer folds into the card's meta. Nothing is written to `~/.claude/tasks`, so a reattached card shows no checklist until its next `update_plan` ‚Äî the plan doesn't survive an app restart (a deliberate v1 tradeoff; the CLI-agnostic tools are in `orchestrator/agentCanvasMcp.ts`).
- **tmux config is minimal by intent**: status off, zero escape-time (Esc is the claude TUI interrupt), `mouse on` for **scrollback only** (the renderer strips mouse-tracking and synthesizes wheel reports itself). The conf is re-sourced on `prepare()` so a running server picks up changes without a fleet restart.
- **`openTerminal`** spawns a *second* tmux client (mobile mirror) via `attach-session` ‚Äî never `new-session`, so a phone can't create a session; killing it just detaches.
- **`paneCommand` is observe-only** (two reads, no mutation) ‚Äî it honors the bare-shell card's no-orchestration contract, walking the pane's tty process table to find the user-typed foreground child.
