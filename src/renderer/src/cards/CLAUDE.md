# cards

The per-card UI for the canvas. A card is one unit of work on the canvas — an `agent` (a watched `claude` CLI in tmux), a bare `shell`, or a `browser` (an in-app Electron `<webview>`). Terminal cards (agent/shell) own a live xterm; browser cards own a live webview guest instead — no tmux/pty/spine session. The focused/master card shows its live surface, and stacked (unfocused) cards overlay a compact "face" on top of the still-mounted surface. This folder also holds the held-ask toast stack that lets you answer permission requests and questions from orbit, and the app's auto-update toast.

## Files

- **CardNode.tsx** — The whole card frame: chrome (border + window bar) wrapping the card's live surface, and the dispatcher between kinds. For terminal kinds it mounts a `TerminalView`; for `browser` a `BrowserView`. The window bar shows kind-appropriate identity (bot/`>_`/page-favicon mark; agent name vs. browser page-title/host) and only agents get the status colour + live status dot — shells and browsers stay neutral. A browser opened by an agent earns two pieces of provenance UI: its window bar's middle slot shows the owner's stated `reason` (in place of a task line), and an `ownerName` chip on the right flies to that agent (`onFlyToOwner`) on click. When `stacked`, overlays a clickable `PosterFace` (agent, fed `browserThumb` when it owns a browser), `ShellFace` (shell), or `BrowserFace` (browser) that promotes on click. Passes `dormant` (webview-budget eviction) through to `BrowserView`.
- **TerminalView.tsx** — Owns the xterm `Terminal` instance and its addons; the one place an agent/shell pty is spawned (via `ensureCard`, after subscribing). Never unmounts across face switches. Only ever runs for agent/shell cards (browsers route to `BrowserView`).
- **PosterFace.tsx** — A stacked *agent* card's compact face: a "mission poster" leading with the task, with the active todo step, a state reason line, a progress bar, subagent count, and attention-debt minutes pinned low. An optional `browserThumb` (the snapshot of the browser this agent owns) shows what its browser is looking at, a glance without promoting.
- **ShellFace.tsx** — A stacked *shell* card's header strip: just the running command pinned to the top, monochrome (colour on this canvas always means an agent needs you). The live terminal behind it *is* the preview.
- **BrowserView.tsx** — A browser card's live surface: the webview analogue of `TerminalView`. Owns an Electron `<webview>` guest (its own process, the shared `persist:browser` session) built imperatively once on mount and never torn down across face switches, plus the address bar + back/forward/reload chrome that drives it. Reports navigation/title/favicon and a demote-time blur snapshot up via `onNavigate`; takes orchestrator-driven `goto` (load on nonce change) alongside user-driven navigation through the bar. On `dom-ready` it fires `window.canvas.browserReady(cardId, webContentsId)` (and `(cardId, null)` on teardown) so main's readiness map can gate browser tools on a real DOM instead of a fixed delay (and Tier-B CDP learns the id). On mount it also registers an imperative control handle in `browserBridge` (`capturePage` screenshot + `executeJavaScript` running the shared driver) for the Tier-A see-and-control fallback. The `dormant` prop is the webview-budget evict: the guest is *not* mounted at all (its process/GL context freed; the snapshot face covers the card) and re-mounts/reloads `url` when it clears — a wake loses in-page scroll/form state but not login (shared partition).
- **browserBridge.ts** — A module-level registry mapping `cardId` → an imperative `BrowserHandle` (`screenshot`/`read`/`act`). The renderer half of the shared `BrowserController`: a browser card's `<webview>` lives here in the renderer, but the orchestrator's drive commands arrive in `Canvas`'s command handler — rather than thread per-card refs through the node tree, `BrowserView` registers a handle on mount (identity-checked unregister on unmount) and the handler looks it up (`getBrowser`). The ref-for-stable-subscription seam, not prop drilling. Tier A drives the webview tag through this handle; a future Tier B moves the same handle to main over CDP without changing the surface.
- **BrowserFace.tsx** — A stacked *browser* card's compact face: the snapshot captured when the card was demoted (the live page stays alive but hidden behind it), or a globe placeholder. No chrome of its own — identity reads from the window bar above.
- **AskToasts.tsx** — Held `PermissionAsk` toasts rising from the bottom: who's asking, their task, and an Allow/Deny gate answered from orbit.
- **QuestionToasts.tsx** — Held `AskUserQuestion` asks as a chooser stack (via the design-system `AskUserQuestions`): renders the questions/options instead of an Allow/Deny gate, injecting the choice on answer.
- **UpdateToast.tsx** — The app's own auto-update toast (bottom-left, away from the per-agent toasts): download progress, then Restart/Later.
- **meta.ts** — `CardMeta`/`CardData` types, the `STATUS_COLORS` palette, the `isLoud` predicate, and `applyCardEvent` — the pure reducer that folds spine `CardEvent`s into a card's accumulated meta. `CardData` also carries the browser-only fields: `url` (live + persisted), transient `title`/`favicon`/`snapshot`, the `goto` nav request, the `onNavigate` callback (whose patch is the shared `BrowserNavPatch` shape — also used by `BrowserView`'s prop and the canvas's `navigateCard`, so a new field can't be added at one site and dropped at the others), and — for a browser opened by an agent — `ownerCardId` + `reason` (the owning agent and its stated purpose), both *persisted* (via `CardRecord`) so the ownership link survives a restart and a re-request finds the same browser rather than spawning a new one.

## Architecture / data flow

**Face switching.** `CardNode` renders exactly one live surface — `TerminalView` for agent/shell, `BrowserView` for browser. The `stacked` prop only toggles which face *composites* over it; the surface (xterm instance or webview guest) is created once on mount and lives for the card's lifetime, so scrollback / page+login+scroll state survive stacking and project switches. A stacked agent hides its terminal (`hidden`) and shows the `PosterFace`; a stacked browser likewise hides its page and shows the snapshot `BrowserFace`; a stacked shell keeps its terminal *visible* as the preview and only lays a `ShellFace` strip on top. Stacked surfaces are made inert (`interactive={false}`) so a drag can't start an xterm selection (or land a click inside the page) instead of clicking through to promote.

**Browser cards (BrowserView).** No tmux/pty/spine — nothing here calls `ensureCard`; the card is just a webview guest. The guest is appended imperatively (so its creation-time `persist:browser` partition can be set before attach) and removed on unmount, which drops its process and listeners. `hidden` toggles CSS *visibility* (not display) so the guest keeps a live offscreen surface; the demote (master→stacked) edge captures a `capturePage` thumbnail fed to `BrowserFace`. The address bar / nav buttons drive the webview directly; the orchestrator drives it through `goto` (loaded each time its nonce advances, seeded so the mount-time `src` isn't double-loaded). All navigation/title/favicon/snapshot changes fold back through `onNavigate` so the canvas can persist the url and refresh the chrome. No `allowpopups` (window.open / target=_blank are dropped in v1), keeping the host's BrowserWindow denial intact.

**See-and-control + dormancy.** A browser card is observable and drivable. On `dom-ready` the view reports readiness up (`browserReady(cardId, webContentsId)`, cleared with `null` on teardown) so main can wait on a real DOM and hand the id to Tier-B CDP; on mount it also registers a `BrowserHandle` (`screenshot`/`read`/`act`) in `browserBridge` for the Tier-A fallback (`capturePage` + `executeJavaScript` of the shared `@shared/browserDriver` set-of-marks read/action scripts). The mount/teardown `useEffect` is keyed on `[cardId, dormant]`: a stacked↔master toggle keeps the same guest (the page survives), but a `dormant` flip rebuilds it. `dormant` is the canvas's webview-budget eviction — the lowest-ranked browsers past the budget mount *no guest at all* (process/GL freed, snapshot face covering them) and re-mount + reload `url` on wake; only in-page scroll/form state is lost (login rides the shared partition).

**xterm mounting (TerminalView).** A single `useEffect` keyed on `cardId`/`folder`/`kind` builds the `Terminal`, loads `FitAddon` and (best-effort) `WebglAddon`, opens it into a ref div, fits once, then subscribes to pty data (`onPtyData`) before calling `ensureCard` so no output byte outruns the listener. A `ResizeObserver` debounces (100 ms) refits so a burst of size changes produces one fit; each fit mirrors cols/rows into the pty via `resize`. A `MutationObserver` on `<html>` re-resolves the terminal theme on dark/light flips. A streaming filter (`makeMouseModeFilter`) strips mouse-tracking DECSET/DECRST sequences from the stream so xterm stays in native-selection mode; the wheel handler synthesizes SGR scroll reports straight to tmux, and typing while scrolled snaps back to live (`leaveScrollback`) before delivering the key.

**Asks/questions → toasts.** Held asks arrive as `PermissionAskInfo` / `QuestionAskInfo` (from `@shared/types`). `AskToasts` and `QuestionToasts` are pure presentational stacks rendered together in a shared bottom overlay (owned by the canvas, not here); each takes a `contextFor(cardId)` to label who's asking. Answering from orbit calls back into the canvas; clicking a toast body flies to the card, which on terminal engage releases the ask to the CLI's native dialog.

**meta.ts.** The canvas owns per-card state and feeds it exclusively through `applyCardEvent` (a stateless, testable reducer). It folds status (stamping `statusSince` on change), detail/task/summary/model/permissionMode, subagent deltas, and todo changes (replace/clear/add/update) into `CardMeta`, which is everything the chrome and poster render.

## Conventions & gotchas

- **Allow/Deny then native dialog.** While an ask is held (toast up), the in-terminal dialog is suppressed. The toast's Allow/Deny answers it remotely; but *engaging the terminal* (`onEngage`, fired on mousedown in `TerminalView`) releases the held ask to the CLI's native on-terminal dialog and clears the toast. Two paths to the same decision.
- **WebGL is best-effort.** `WebglAddon` is tried in a try/catch with a DOM-renderer fallback "fine for a handful of cards" — be mindful of the GL context budget if many terminals mount at once. It disposes on context loss.
- **Resize is debounced for the agent's sake**, not just perf: every fit SIGWINCHes the pty and the `claude` TUI reflows, so an undebounced 60 Hz resize thrashes the agent. The mount-time fit stays immediate so the pty spawns correctly sized.
- **Non-agent cards are monochrome and silent.** `shell` and `browser` cards have no agent to speak for them, so both use neutral `var(--border)` chrome with no status dot or pulse (colour on this canvas always means an agent needs you). Shells follow their pane's cwd/command (via the `useShellTitles` poll passed as `title`); browsers follow their live page title/host. The spine never speaks for either.
- **Closing a browser card is session-less.** Unlike agent/shell cards it kills no tmux session — there's only the webview guest to drop. (See Canvas for the kind-aware close path.)
- **Owner link is computed upstream, surfaced here.** `CardNode` only *renders* provenance — `ownerName`, `onFlyToOwner`, and an agent's `browserThumb` are derived by `Canvas` from the `ownerCardId` on each browser's `CardData`; the persisted source of truth is `ownerCardId` + `reason` in `meta.ts`. Cards don't resolve ownership themselves.
- **Toast buttons are plain `<button>`, not the design-system `Button`**, so Allow/Deny keep their go/stop status background colors (the shared Button paints bg from a variant and would mask a `bg-*` class).
- Closing a card kills its tmux session (`onClose`) — it's destructive, not just a hide.
