# cards

The per-card UI for the canvas. A card is one unit of work on the canvas — an `agent` (a headless `claude`/`codex` session, no terminal), a `shell` (a bare `$SHELL` in a direct pty, no tmux), or a `browser` (an in-app Electron `<webview>`). Only shell and browser cards own a live, always-mounted surface (xterm / webview guest); an agent's live surface — a transcript + composer — mounts only while it's the master, since it's cheap React state to reconstruct from a `CardEvent`/`TranscriptItem` replay, unlike an xterm's scrollback or a webview's page state. The focused/master card shows its live surface, and stacked (unfocused) cards overlay a compact "face" on top of it (or, for a stacked agent, in place of nothing — see below). This folder also holds the held-ask toast stack that lets you answer permission requests and questions from orbit, and the app's auto-update toast.

## Files

- **CardNode.tsx** — The whole card frame: chrome (border + window bar) wrapping the card's live surface, and the three-way dispatcher between kinds. `browser` → `BrowserView`; `shell` → the (now shell-only) `TerminalView`; `agent` → `TranscriptView` (which renders its own composer via assistant-ui's `Thread`), but ONLY when `!stacked` — a stacked agent renders nothing live at all, just the `PosterFace` overlay button. Also runs the agent-start effect: `useEffect(() => { if (isAgent) void window.canvas.startAgent(id, folder, data.cli) }, [id, folder, data.cli, isAgent])` — idempotent (a no-op if the session's already registered) and NOT gated on the transcript being visible, so a freshly spawned STACKED agent still starts working immediately. The window bar shows kind-appropriate identity (bot/`>_`/page-favicon mark; agent name vs. browser page-title/host) and only agents get the status colour + live status dot — shells and browsers stay neutral. No BYPASS chip anymore (every card runs bypass now — a permanent badge would just be noise). A browser opened by an agent earns two pieces of provenance UI: its window bar's middle slot shows the owner's stated `reason` (in place of a task line), and an `ownerName` chip on the right flies to that agent (`onFlyToOwner`) on click. When `stacked`, overlays a clickable `PosterFace` (agent, fed `browserThumb` when it owns a browser), `ShellFace` (shell), or `BrowserFace` (browser) that promotes on click. Passes `dormant` (webview-budget eviction) through to `BrowserView`.
- **TranscriptView.tsx** — An agent card's live surface: a headless session has no pty, so this replaces the terminal entirely. The DATA layer is unchanged from the old hand-rolled view: self-subscribes to `onTranscriptItem` BEFORE calling `loadTranscript` (buffering any push that lands mid-load, merged last-wins-by-id once the backlog resolves) — the same "no byte/item can outrun the listener" discipline `TerminalView` uses for `onPtyData`; state is a plain array upserted by `TranscriptItem.id`. The PRESENTATION is now bridged into `@assistant-ui/react`'s `useExternalStoreRuntime`: `groupItems` coalesces a turn's items into one assistant-ui message per role (a `user` item starts its own group; every assistant-side item — text, tool calls, errors — folds into ONE open assistant group until the next `user`/`turn` boundary, which is what lets consecutive tool calls render as a single grouped card, since assistant-ui only groups parts *within* a message), `toMessage`/`toPart` map a group onto `ThreadMessageLike`, and `Thread` (`components/assistant-ui/thread.tsx`) owns the whole rendered surface — message list, markdown, tool cards, auto-scroll, AND the composer (no separate `Composer.tsx` anymore). Sending routes back out through the runtime's `onNew` → `stripDirectives` → `sendToCard`; the resulting `user` item is echoed by main, so nothing is appended optimistically. `onCancel` wires the stop button to `interruptCard`.
- **composerTriggers.tsx** — The composer's `/` (skills, via `listSkills`) and `@` (repo files, via `searchFiles` → `git.ts`'s `listRepoFiles`) trigger pickers, rendered by `CardComposerTriggers` inside assistant-ui's `Unstable_TriggerPopoverRoot`. `CardChatContext` (provided by `TranscriptView`) carries the card's `cli` (indexes `CLI_SKILL_PREFIX` from `@shared/types` for the skill-invocation prefix — the same record each driver's `skillRef()` reads, so renderer and spine can't drift) and `folder` (to scope the file search) down to the generic, vendored trigger-popover component. A pick inserts a `:type[label]{name=id}` directive, rendered inline as a chip by `DirectiveChip`; `stripDirectives` rewrites it back to the clean `id` (invocation string or file path) before the text reaches the CLI, so the agent never sees chip syntax.
- **TerminalView.tsx** — Shell cards ONLY now (agent cards never mount this — see `TranscriptView` above). Owns the xterm `Terminal` instance and its addons; the one place a shell's direct pty is spawned (via `ensureShell`, after subscribing to `onPtyData`). A direct pty (no tmux) means this is "a normal terminal" with nothing to strip or synthesize — no mouse-mode filtering, no SGR wheel synthesis, no `leaveScrollback` round-trip: xterm's own local scrollback and native mouse handling just work. Takes only `{cardId, folder, interactive}` (no `kind`/`cli`/`hidden`/`onEngage` — those all died with the agent-terminal path).
- **PosterFace.tsx** — A stacked *agent* card's compact face: a "mission poster" leading with the task, with the active todo step, a state reason line, a progress bar, subagent count, and attention-debt minutes pinned low. An optional `browserThumb` (the snapshot of the browser this agent owns) shows what its browser is looking at, a glance without promoting.
- **ShellFace.tsx** — A stacked *shell* card's header strip: just the running command pinned to the top, monochrome (colour on this canvas always means an agent needs you). The live terminal behind it *is* the preview.
- **BrowserView.tsx** — A browser card's live surface: the webview analogue of `TerminalView`. Owns an Electron `<webview>` guest (its own process, the shared `persist:browser` session) built imperatively once on mount and never torn down across face switches, plus the address bar + back/forward/reload chrome that drives it. Reports navigation/title/favicon and a demote-time blur snapshot up via `onNavigate`; takes orchestrator-driven `goto` (load on nonce change) alongside user-driven navigation through the bar. On `dom-ready` it fires `window.canvas.browserReady(cardId, webContentsId)` (and `(cardId, null)` on teardown) so main's readiness map can gate browser tools on a real DOM instead of a fixed delay (and Tier-B CDP learns the id). On mount it also registers an imperative control handle in `browserBridge` (`capturePage` screenshot + `executeJavaScript` running the shared driver) for the Tier-A see-and-control fallback. The `dormant` prop is the webview-budget evict: the guest is *not* mounted at all (its process/GL context freed; the snapshot face covers the card) and re-mounts/reloads `url` when it clears — a wake loses in-page scroll/form state but not login (shared partition). Unchanged by the headless-substrate rewrite.
- **browserBridge.ts** — A module-level registry mapping `cardId` → an imperative `BrowserHandle` (`screenshot`/`read`/`act`). The renderer half of the shared `BrowserController`: a browser card's `<webview>` lives here in the renderer, but the orchestrator's drive commands arrive in `Canvas`'s command handler — rather than thread per-card refs through the node tree, `BrowserView` registers a handle on mount (identity-checked unregister on unmount) and the handler looks it up (`getBrowser`). The ref-for-stable-subscription seam, not prop drilling. Tier A drives the webview tag through this handle; a future Tier B moves the same handle to main over CDP without changing the surface.
- **BrowserFace.tsx** — A stacked *browser* card's compact face: the snapshot captured when the card was demoted (the live page stays alive but hidden behind it), or a globe placeholder. No chrome of its own — identity reads from the window bar above.
- **AskToasts.tsx** — Held `PermissionAsk` toasts rising from the bottom: who's asking, their task, and an Allow/Deny gate answered from orbit. Dormant in practice now — headless cards run unattended (bypass), so there's no held permission ask left to hold; kept wired (harmless) rather than ripped out mid-rewrite.
- **QuestionToasts.tsx** — Held `ask_user` questions (the canvas MCP tool — every card's structured-question channel, unaffected by the headless rewrite) as a chooser stack (via the design-system `AskUserQuestions`): renders the questions/options instead of an Allow/Deny gate, resolving the held tool call on answer. This is now the ONLY toast that sees live traffic.
- **UpdateToast.tsx** — The app's own auto-update toast (bottom-left, away from the per-agent toasts): download progress, then Restart/Later.
- **meta.ts** — `CardMeta`/`CardData` types, the `STATUS_COLORS` palette, the `isLoud` predicate, and `applyCardEvent` — the pure reducer that folds spine `CardEvent`s into a card's accumulated meta (unchanged by the rewrite — `CardEvent`'s shape and the fold logic didn't need to change). `CardData` carries `role` (the Mastermind role — planner/lead/worker — driving the issue-MCP tool grant; absent = plain worker) and `cli` (which CLI — claude/codex — backs the card, chosen at spawn and passed to `startAgent`; absent = claude), both persisted via `CardRecord`. `CardMeta.sessionId` is now the CLI's own session/thread id (not a tmux name) — a relaunched card's first send resumes the same conversation from it. `CardData.onEngage` is GONE (there's no terminal to engage for an agent card, and shells never held asks). It also carries the browser-only fields: `url` (live + persisted), transient `title`/`favicon`/`snapshot`, the `goto` nav request, the `onNavigate` callback (whose patch is the shared `BrowserNavPatch` shape — also used by `BrowserView`'s prop and the canvas's `navigateCard`, so a new field can't be added at one site and dropped at the others), and — for a browser opened by an agent — `ownerCardId` + `reason` (the owning agent and its stated purpose), both *persisted* (via `CardRecord`) so the ownership link survives a restart and a re-request finds the same browser rather than spawning a new one.

## Architecture / data flow

**Face switching.** `CardNode` renders exactly one live surface per kind — `BrowserView` for browser, `TerminalView` for shell, `TranscriptView` (composer included) for agent (master only). Browser/shell surfaces are created once on mount and live for the card's lifetime (`stacked` only toggles which face *composites* over them), so a webview's page state / a shell's scrollback survive stacking and project switches — that's the reason those two stay always-mounted. An agent's transcript does NOT need that trick: it's just React state rebuilt from `loadTranscript` + `onTranscriptItem` on mount, so a stacked agent mounts nothing at all and the `PosterFace` overlay button is the whole card. A stacked browser hides its page (`hidden`) and shows the snapshot `BrowserFace`; a stacked shell keeps its terminal *visible* as the preview and only lays a `ShellFace` strip on top. Stacked shell/browser surfaces are made inert (`interactive={false}`) so a drag can't start an xterm selection (or land a click inside the page) instead of clicking through to promote.

**Browser cards (BrowserView).** No pty/spine session — nothing here calls into the spine at all; the card is just a webview guest. The guest is appended imperatively (so its creation-time `persist:browser` partition can be set before attach) and removed on unmount, which drops its process and listeners. `hidden` toggles CSS *visibility* (not display) so the guest keeps a live offscreen surface; the demote (master→stacked) edge captures a `capturePage` thumbnail fed to `BrowserFace`. The address bar / nav buttons drive the webview directly; the orchestrator drives it through `goto` (loaded each time its nonce advances, seeded so the mount-time `src` isn't double-loaded). All navigation/title/favicon/snapshot changes fold back through `onNavigate` so the canvas can persist the url and refresh the chrome. No `allowpopups` (window.open / target=_blank are dropped in v1), keeping the host's BrowserWindow denial intact.

**See-and-control + dormancy.** A browser card is observable and drivable. On `dom-ready` the view reports readiness up (`browserReady(cardId, webContentsId)`, cleared with `null` on teardown) so main can wait on a real DOM and hand the id to Tier-B CDP; on mount it also registers a `BrowserHandle` (`screenshot`/`read`/`act`) in `browserBridge` for the Tier-A fallback (`capturePage` + `executeJavaScript` of the shared `@shared/browserDriver` set-of-marks read/action scripts). The mount/teardown `useEffect` is keyed on `[cardId, dormant]`: a stacked↔master toggle keeps the same guest (the page survives), but a `dormant` flip rebuilds it. `dormant` is the canvas's webview-budget eviction — the lowest-ranked browsers past the budget mount *no guest at all* (process/GL freed, snapshot face covering them) and re-mount + reload `url` on wake; only in-page scroll/form state is lost (login rides the shared partition).

**Shell terminal mounting (TerminalView).** A single `useEffect` keyed on `cardId`/`folder` builds the `Terminal`, loads `FitAddon` and (best-effort) `WebglAddon`, opens it into a ref div, fits once, then subscribes to pty data (`onPtyData`) before calling `ensureShell` so no output byte outruns the listener. A `ResizeObserver` debounces (100 ms) refits so a burst of size changes produces one fit; each fit mirrors cols/rows into the pty via `resize`. A `MutationObserver` on `<html>` re-resolves the terminal theme on dark/light flips. Since the pty is direct (no tmux), input goes straight to `window.canvas.write` and output straight to `term.write` — no mouse-mode stripping, no SGR wheel synthesis, no scrollback round-trip.

**Agent transcript mounting (TranscriptView).** No `useEffect`-driven spawn here at all — the agent's session is started by `CardNode`'s effect (see above), independent of whether the transcript is even mounted. `TranscriptView`'s own `useEffect` (keyed on `cardId`) only concerns itself with REPLAYING that session's history: subscribe to `onTranscriptItem`, buffer anything that arrives before `loadTranscript` resolves, then merge (last-wins by id) and go live. That item array feeds `useExternalStoreRuntime` (grouped + mapped, see the TranscriptView file note above), and `Thread` renders from the runtime — message list, composer, and trigger pickers (`composerTriggers.tsx`) all in one vendored surface. Promoting a stacked agent to master simply mounts a fresh `TranscriptView`, which replays the same way — cheap, because there's no GL context or scrollback buffer to rebuild, just a JSON array.

**Asks/questions → toasts.** Held asks arrive as `PermissionAskInfo` / `QuestionAskInfo` (from `@shared/types`). `AskToasts` and `QuestionToasts` are pure presentational stacks rendered together in a shared bottom overlay (owned by the canvas, not here); each takes a `contextFor(cardId)` to label who's asking. Answering from orbit calls back into the canvas; there is no terminal-engage fallthrough anymore (that died with `onEngage`) — a held ask/question is released only by forward progress, the card's session ending, or an explicit orbit decision.

**meta.ts.** The canvas owns per-card state and feeds it exclusively through `applyCardEvent` (a stateless, testable reducer, untouched by the rewrite). It folds status (stamping `statusSince` on change), detail/task/summary/model/permissionMode, subagent deltas, and todo changes (replace/clear — the plan flows through the canvas MCP `update_plan`, which replaces wholesale) into `CardMeta`, which is everything the chrome and poster render. The new `TranscriptItem` feed is a SEPARATE, parallel channel (`onTranscriptItem`/`loadTranscript`) that `TranscriptView` owns directly — it does not go through `CardMeta`/`applyCardEvent`.

## Conventions & gotchas

- **Toasts answer from orbit only — no native-dialog fallthrough.** There's no terminal for an agent card to release a held ask/question into anymore, so Allow/Deny/answer/decline are the only paths; a held item otherwise clears on forward progress or the card's session ending (`onSessionEnded` for agents, `onPtyExit` for shells).
- **WebGL is shell-only now, and rare.** `WebglAddon` (best-effort, DOM-renderer fallback) only ever loads for a `shell` card's `TerminalView` — agent cards have no xterm/GL context at all anymore, which is the real fix for the old ~16-context Chromium ceiling: it now only has to stretch across shells + browsers, not every agent too.
- **Resize is debounced for the shell's sake**, not just perf: every fit SIGWINCHes the pty, so an undebounced 60 Hz resize thrashes it. The mount-time fit stays immediate so the pty spawns correctly sized. (Irrelevant to agent cards — there's no pty to SIGWINCH.)
- **Non-agent cards are monochrome and silent.** `shell` and `browser` cards have no agent to speak for them, so both use neutral `var(--border)` chrome with no status dot or pulse (colour on this canvas always means an agent needs you). Shells follow their pane's cwd/command (via the `useShellTitles` poll passed as `title`, now backed by a ps-walk of the direct pty's pid, not a tmux pane query); browsers follow their live page title/host.
- **Closing a browser card is session-less.** Unlike agent/shell cards it kills no session — there's only the webview guest to drop. (See Canvas for the kind-aware close path.) Closing an agent card ends its headless session (SDK query / codex subprocess); closing a shell kills its direct pty.
- **Owner link is computed upstream, surfaced here.** `CardNode` only *renders* provenance — `ownerName`, `onFlyToOwner`, and an agent's `browserThumb` are derived by `Canvas` from the `ownerCardId` on each browser's `CardData`; the persisted source of truth is `ownerCardId` + `reason` in `meta.ts`. Cards don't resolve ownership themselves.
- **Toast buttons are plain `<button>`, not the design-system `Button`**, so Allow/Deny keep their go/stop status background colors (the shared Button paints bg from a variant and would mask a `bg-*` class).
- Closing a card ends its session (`onClose`) — it's destructive, not just a hide.
